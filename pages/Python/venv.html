1. What a virtual environment actually is

When you run:

source venv/bin/activate

you’re just changing your shell environment (mainly the $PATH) so that Python and pip commands point to the 
Python interpreter and site-packages inside that venv, instead of the system-wide ones.

Example:

which python
# -> /path/to/your/project/venv/bin/python


---


2. How for ex yt-dlp is managed inside a venv

When you install it inside the venv:

pip install yt-dlp

it’s stored locally in that venv directory under:

venv/lib/pythonX.Y/site-packages/


That version is completely independent from any system packages or other virtual environments.

---


3. How to update it

To update yt-dlp (or any Python package inside the venv), you must either activate the venv or call its pip explicitly.

Using the activated venv:

source venv/bin/activate
pip install --upgrade yt-dlp


Without activating:

/path/to/venv/bin/pip install --upgrade yt-dlp


---


4. What about apt upgrade?

apt manages system-level packages installed via Debian/Ubuntu repositories 
(e.g. /usr/bin/python3, /usr/lib/python3/dist-packages/, etc).

It does not touch or even know about anything installed inside your virtual environments.

So:

    
  
  • Running apt upgrade while inside or outside the venv makes no difference.
    • 
  
  • Your venv packages (like yt-dlp) will not be upgraded by apt.
    •

6. Optional: Automating venv package updates

You can make a mini “apt upgrade” for your venv using:

source venv/bin/activate
pip list --outdated --format=freeze | cut -d = -f 1 | xargs -n1 pip install -U


This command:

    
  
  • Lists all outdated packages
    • 
  
  • Extracts their names
    • 
  
  • Upgrades them automatically inside the active venv
